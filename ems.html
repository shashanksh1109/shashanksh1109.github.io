Layer 1

Embedded firmware

01
Embedded C — Sensor Firmware & PID Controllers
ARM Cortex-M4 · FreeRTOS · ESP32 · CAN/SPI/I2C/UART · DMA
▼

What it does

Multi-node embedded firmware simulating a real smart building — temperature sensors per zone, PID-controlled HVAC actuators, and power metering units. Each node runs concurrently using FreeRTOS task scheduling, sending structured binary packets over TCP to the Python gateway.

FreeRTOS task architecture

  • Sensor task (priority 3): DMA-driven ADC sampling at 1kHz — 40% CPU load reduction vs polling
  • PID task (priority 4): Runs every 10ms, reads sensor buffer, computes control output
  • Comms task (priority 2): Packs struct, sends TCP packet, handles retries
  • Watchdog task (priority 1): Kicks IWDG every 1s — system-level safety net
  • Semaphores between sensor and PID tasks; message queues for comms handoff

Hardware abstraction layer

  • HAL interface allows swapping between simulated and real ESP32/DHT22 hardware
  • Runtime flag: --hardware activates physical sensor reads
  • Same firmware binary runs on both — no recompilation needed

PID thermal control

  • Ziegler–Nichols tuning: identified Ku and Tu from sustained oscillation on oscilloscope
  • Anti-windup: conditional integration stops accumulation during actuator saturation
  • 20% improvement in temperature regulation accuracy vs manual tuning
  • 15% reduction in thermal overshoot
C PID compute with anti-windup 
float pid_compute(PID_t *pid, float sp, float pv) {
  float error = sp - pv;
  float deriv = (error - pid->prev_err) / pid->dt;

  // Anti-windup: only integrate if not saturated
  if (!pid->saturated)
    pid->integral += error * pid->dt;

  // Clamp integral term
  pid->integral = clamp(pid->integral,
                        -pid->int_max, pid->int_max);

  pid->prev_err = error;
  return pid->Kp * error
       + pid->Ki * pid->integral
       + pid->Kd * deriv;
}
⚡ Design Decision: Why FreeRTOS over bare-metal?

The system has 4 concurrent tasks with different timing requirements: sensor polling at 100ms, PID at 10ms, TCP comms at variable latency, and analytics every 60s. FreeRTOS priority scheduling guarantees the PID and sensor tasks always meet their deadlines regardless of TCP load — something a bare-metal polling loop cannot cleanly guarantee. The overhead is ~12µs per context switch on Cortex-M4 — acceptable at these frequencies.

Layer 2

The binary protocol

02
Custom 18-Byte Binary Messaging Protocol
Struct-packed · CRC-validated · Little-endian · TCP transport
▼

Packet layout

B0device_id
B1sensor_type
B2–B5timestamp_ms (uint32)
B6–B9value (float)
B10–B13setpoint (float)
B14–B15flags (uint16)
B16–B17CRC16 ✓
Total: 18 bytes · __attribute__((packed)) · no padding
⚡ Design Decision: Why binary over JSON?

Bandwidth: 18 bytes vs ~80 bytes for equivalent JSON — a 4.4× reduction. At 16 sensors polling at 10Hz, that is 16 × 10 × 80 = 12.8KB/s for JSON vs 2.88KB/s for binary. On UART at 115200 baud, JSON consumes 88% of bandwidth; binary consumes 20%.

Parsing determinism: A binary struct cast is O(1) and takes the same time every call. JSON parsing is O(n) string scanning — non-deterministic in a real-time control loop.

Memory: No dynamic string allocation on the microcontroller. The entire packet lives in a statically-declared struct — essential with 64KB RAM.

C Struct definition + serialization 
__attribute__((packed))
typedef struct {
  uint8_t  device_id;     // 1 byte
  uint8_t  sensor_type;   // 1 byte
  uint32_t timestamp_ms;  // 4 bytes
  float    value;         // 4 bytes — sensor reading
  float    setpoint;      // 4 bytes — PID target
  uint16_t flags;         // 2 bytes — status bits
  uint16_t checksum;      // 2 bytes — CRC16
} SensorPacket_t;           // Total: 18 bytes

// Send: cast struct directly to byte array — zero-copy
SensorPacket_t pkt = { .device_id = 1, .value = temp };
pkt.checksum = crc16((uint8_t*)&pkt, sizeof(pkt) - 2);
tcp_send(sock, (uint8_t*)&pkt, sizeof(pkt));
Python Gateway parser — O(1) struct unpack 
import struct

PACKET_FMT = '   # little-endian, 18 bytes
PACKET_SIZE = struct.calcsize(PACKET_FMT)  # = 18

def parse_packet(raw: bytes) -> dict:
    fields = struct.unpack(PACKET_FMT, raw)
    device_id, sensor_type, ts, val, sp, flags, crc = fields

    # Validate CRC before processing
    expected = crc16(raw[:-2])
    if crc != expected:
        raise ValueError(f"CRC mismatch: {crc} != {expected}")

    return {"device_id": device_id, "value": val,
            "timestamp_ms": ts, "setpoint": sp}
Layer 3

Gateway & backend

03
Python Gateway · Spring Boot API · PostgreSQL · Redis
Multi-threaded TCP · MQTT · wolfSSL · JWT · WebSocket · JPA/Hibernate
▼

Python IoT gateway

  • Multi-threaded TCP server: one thread per sensor node connection
  • Circular buffer (1000 entries) for lock-free producer-consumer ingestion
  • Analytics engine: mean/stddev/min/max per zone every 60s
  • MQTT pub/sub integration alongside TCP for device management
  • wolfSSL AES-TLS on all sensor links — encrypted at transport layer
  • Persistent writes to PostgreSQL via db_writer.py with connection pooling

PostgreSQL time-series schema

  • sensor_readings — raw append-only time-series
  • zones / devices — building topology
  • hvac_events — actuator state changes
  • analytics_summary — pre-computed aggregates for fast dashboard queries
  • Composite index on (zone_id, recorded_at DESC) for sub-100ms range queries
  • Redis cache-aside for analytics layer — 5-min TTL, invalidated on new readings

Spring Boot REST API

  • Layered MVC: Controller → Service → Repository — no business logic in controllers
  • JWT auth middleware on all endpoints — stateless, 15-min expiry
  • WebSocket STOMP server — real-time push to React dashboard on every new sensor reading
  • JPA/Hibernate ORM with optimized query design — DTO projection prevents over-fetching
  • Global @ControllerAdvice exception handler — maps all errors to structured JSON responses
🐛 Real Bug: The 0.0.0.0 vs 127.0.0.1 Fix

Symptom: After AWS deployment with Docker Compose, the dashboard showed no live data. All containers healthy. No errors in logs.

Root cause: The Python gateway was bound to 127.0.0.1. In Docker's bridge network, each container has its own loopback — sensor containers could not reach the gateway's loopback interface.

Fix: Changed bind address to 0.0.0.0 to accept on all interfaces including the Docker bridge network IP. Data flowed immediately.

AWS infrastructure

AWS Infrastructure · Terraform IaC · GitHub Actions CI/CD

Terraform module structure

  • modules/vpc — VPC, public/private subnets, IGW, NAT, route tables
  • modules/compute — EC2 Launch Template, ASG, ALB, target groups
  • modules/database — RDS PostgreSQL multi-AZ, subnet group, parameter group
  • modules/monitoring — CloudWatch alarms, dashboards, SNS topics
  • Remote state: S3 bucket + DynamoDB locking — no concurrent apply conflicts

VPC security architecture

  • ALB SG: accepts 80/443 from internet only
  • EC2 SG: accepts 8080 from ALB SG only — never from internet
  • RDS SG: accepts 5432 from EC2 SG only — completely private
  • IAM instance role: read S3 (logs), write CloudWatch metrics, read SSM Parameter Store (DB password) — nothing else

CI/CD pipeline — 7 stages

  • Stage 1 Build: Maven (Java), pip + syntax (Python), gcc (C)
  • Stage 2 Test: JUnit + Mockito, Pytest with 80% coverage threshold
  • Stage 3 Analysis: SpotBugs (Java), pylint (Python), cppcheck (C)
  • Stage 4 Docker: Build images, push to Amazon ECR
  • Stage 5 AMI: Packer bakes new AMI with app pre-installed — cold start in 60s
  • Stage 6 Deploy: Terraform updates Launch Template; ASG rolling refresh — zero downtime
  • Stage 7 Smoke Test: Hit /actuator/health, verify WebSocket, confirm data flow
⚡ Baked AMI vs user data scripts

User data runs at every launch — 5 minutes before healthy. Packer pre-installs everything: new instances launch and are ready in 60 seconds. No runtime dependency on external package repos. The 90% ops effort reduction comes from eliminating 25–30 minutes of manual SSH deployment work per release.
